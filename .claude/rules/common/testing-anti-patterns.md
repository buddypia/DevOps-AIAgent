# Testing Anti-Patterns Rules

## ID: R-CM-011
## Severity: critical
## Enforced by: coverage-threshold-guard（本プロジェクトでは未配備 — prompt-level）

### Iron Laws

```
1. モックの動作をテストしない
2. プロダクションクラスにテスト専用メソッドを追加しない
3. 依存関係を理解せずにモックしない
```

### Rules

1. **モック動作テスト禁止**: モック要素に assert しない。実際のコンポーネントの動作をテストするか、モックを除去する
2. **テスト専用メソッド禁止**: プロダクションクラスに、テストからのみ呼び出されるメソッド（destroy, reset など）を追加しない。テストユーティリティに分離する
3. **理解なきモッキング禁止**: モックする前に必ず (1) 実メソッドのサイドエフェクトを確認 (2) テストがそのサイドエフェクトに依存するか確認 (3) 最小限のレベルでモックする
4. **不完全モック禁止**: 実際の API レスポンスの**全体**構造をミラーリングする。現在のテストで使うフィールドだけでなく、ダウンストリームのコードが消費しうるすべてのフィールドを含める
5. **後追いテストの境界**: 実装後にテストを書くと即座にパスする → 何も証明できない。TDD（テスト先行）が基本である

### Gate Functions

**モック要素に assert する前:**
```
質問: 「実際のコンポーネントの動作をテストしているのか、モックの存在をテストしているのか?」
モックの存在テスト → 止めて、assertion を削除するかモックを除去する
```

**プロダクションクラスにメソッドを追加する前:**
```
質問: 「このメソッドはテストでのみ使われるのか?」
YES → 追加しない。テストユーティリティに配置する
```

**メソッドをモックする前:**
```
1. 実メソッドのサイドエフェクトは何か?
2. このテストはそのサイドエフェクトに依存するか?
3. 依存する場合 → より低いレベルでモックする（遅い/外部の処理のみ）
4. 確信が持てない場合 → まず実装で実際にテストを実行して観察する
```

### Red Flags

- `*-mock` のテスト ID に対する assertion
- テストファイルからのみ呼び出されるメソッド
- モックのセットアップがテストの 50% 以上
- モックを除去するとテストが失敗する
- なぜモックが必要なのか説明できない
- 「安全のために」モックする

### TDD がこのアンチパターンを防ぐ理由

1. **テストを先に書く** → 実際に何をテストするのかを考えることを強制する
2. **失敗を確認する** → テストが（モックではなく）実際の動作をテストしているか確認する
3. **最小実装** → テスト専用メソッドが入り込む余地がない
4. **実際の依存関係** → モックする前に、テストが実際に何を必要とするかを確認する
